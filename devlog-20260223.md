### Phase 0 Completion: Device Flashing & USB Access

**Problem**: After flashing Debian firmware, the device was invisible to the Mac — no USB, no ADB, no network.

**Root cause chain**:
1. Debian's `setup_usbgadget.sh` included `ffs.adb` (FunctionFS for ADB), but no `adbd` binary existed on Debian
2. FunctionFS requires a userspace daemon to open endpoints before UDC can attach
3. Without adbd, the USB gadget driver failed to attach → device invisible
4. Even after copying `adbd` from stock Buildroot, it failed because stock is **armhf (32-bit)** but Debian is **aarch64 (64-bit)**
5. macOS also doesn't support RNDIS well — interface appears but link stays "inactive"

**Solution**:
1. Copied armhf shared libraries from stock Buildroot to Debian (`/lib/arm-linux-gnueabihf/`, `/usr/lib/arm-linux-gnueabihf/`)
2. Copied armhf dynamic linker (`/lib/ld-linux-armhf.so.3`)
3. Created symlinks at `/lib/libc.so.6` etc. pointing to armhf copies (safe because aarch64 Debian uses `/lib/aarch64-linux-gnu/` paths)
4. Rewrote `setup_usbgadget.sh` without `set -e`, with error logging to `/tmp/usbgadget.log`
5. Result: ADB works on Debian from Mac

**U-Boot env management**:
- `--send_env` + `env save` is **persistent** (writes to eMMC env partition)
- `pick_boot_slot=run boot_slot_debian;` → always boots Debian
- `pick_boot_slot=run boot_slot_normal;` → always boots stock Buildroot
- To switch: burn mode (buttons 1+4) → `--send_env` → `--continue_boot`

**Device state**:
- Debian 11 (Bullseye), aarch64, kernel 4.9.113
- ADB access working (serial 12345678)
- Input devices: event0 (gpio-keys/buttons), event1 (rotary), event2 (vkeypad), event3 (touchscreen)
- Python 3.9.2 available
- SSH loaded but not starting (privilege separation dir issue, non-critical)

### Key Lessons
- macOS RNDIS support is broken — always prefer ADB or CDC ECM
- FunctionFS (ffs.adb) requires active userspace daemon before UDC attach
- Stock Buildroot is armhf, Debian is aarch64 — need multiarch compat for stock binaries
- `set -e` in boot scripts is dangerous — one failure kills USB gadget setup
- `--send_env` with `env save` is persistent, not session-only

### BLE HID Pairing with macOS (SUCCESS)

**Key discovery**: `btattach -P bcm -B /dev/ttyS1` is the correct approach (not `hciattach any`):
- `hciattach any` creates HCI 4.2 adapter WITHOUT firmware — radio barely works
- `btattach -P bcm` uploads Cypress firmware: HCI upgrades 4.2→5.2, BD address changes to 30:E3:D6:05:AA:CD

**Classic BT HID failed**: macOS doesn't pair with Classic BT HID (sdptool KEYB + L2CAP) — needs actual PSM listeners. Switched to BLE HOGP.

**BLE HID GATT server** (ble_hid_gatt.py):
- Python D-Bus GATT server with HID Service (0x1812)
- Characteristics: HID Information, Report Map, Control Point, Report (notify), Protocol Mode
- BLE Advertisement as peripheral keyboard (Appearance 0x03C1)
- UNIX socket IPC at `/tmp/spotifone_hid.sock` for key injection

**macOS pairing requirements** (critical findings):
1. `hciconfig hci0 noscan` — disable Classic BT, BLE-only
2. `btmgmt --index 0 io-cap 1` — DisplayOnly capability (macOS rejects NoInputNoOutput for HID)
3. `btmgmt --index 0 advertising on` — enable BLE advertising at management level

### Voice Setting Correction

**Previous (wrong) analysis**: Decoded voice=0x0060 as 8-bit 1's complement — concluded S16_LE was mismatched.

**Corrected decode** (Bluetooth Core Spec bit layout):
```
0x0060 = 0000 0000 0110 0000
Bit [1:0]  Input Coding:     00 → Linear PCM
Bit [2]    Reserved:         0
Bit [4:3]  Input Data Format: 00 → 1's complement
Bit [5]    Input Sample Size: 1  → 16-bit
Bit [7:6]  PCM bit padding:  01 → 1 bit MSB pad
Bit [9:8]  Air Coding:       00 → CVSD
```

**Conclusion**: voice=0x0060 IS compatible with S16_LE. The BT controller handles CVSD encoding from 16-bit PCM internally. Reverted voice setting from 0x0014 back to 0x0060. Removed debug dump code.

### SCO 8-bit Fix (SUCCESS)

**Root cause**: Despite voice=0x0060 specifying 16-bit PCM, the BCM4345C0 controller and macOS treat SCO data as **8-bit samples**. Writing 64 bytes of S16_LE (32 samples) to the SCO socket resulted in 64 "samples" at the receiver — audio played at half speed.

**Fix**: Convert S16_LE to signed 8-bit before SCO write:
1. Read 2×MTU bytes from arecord (MTU S16_LE samples)
2. Apply MIC_GAIN (32×) to S16_LE samples
3. Convert: `sco_buf[i] = (unsigned char)((pcm[i] >> 8) & 0xFF)`
4. Write MTU bytes of 8-bit samples to SCO socket

**Result**: Audio plays at correct speed on Mac. Voice is clear and recognizable.

### SCO Audio Distortion — Timing Analysis & Fix

**Problem**: S16_LE direct-to-SCO with SO_SNDBUF=32768 produces correct speed (~16,000 B/s) but distorted audio. Local mic recordings (arecord → WAV) sound clean. Distortion introduced between pipe read and Mac reception.

**Root cause hypothesis**: arecord delivers PCM in ALSA period-sized bursts (e.g. 1024 bytes every 64ms). Our fread(64) drains each burst in ~16 back-to-back writes (all within ~2ms), then nothing for ~60ms. With SO_SNDBUF=32768, write() never blocks, so all 16 packets queue instantly. The BCM4345C0 CVSD encoder FIFO gets overwhelmed during bursts and starved during gaps.

**Fix implemented** (mic_bridge.c `audio_thread_func()`):

1. **Timer-based write pacing**: `clock_nanosleep(CLOCK_MONOTONIC, TIMER_ABSTIME)` ensures exactly one SCO packet per write period (4ms for MTU=64). After each write, timer advances by `n * 62500` ns (= `n / 16000` sec). If timer falls behind (fread blocked), resets to now (no burst catch-up).

2. **S16_LE alignment guard**: After fread(), force even byte count with `n &= ~(size_t)1` to prevent sample misalignment from partial pipe reads.

3. **Enhanced diagnostics**: Periodic log now includes partial read count, odd byte count, and pacing status.

**Corrected lesson 28**: S16_LE IS correct for voice=0x0060. Previous "8-bit" conclusion was based on a different symptom (half-speed from SO_SNDBUF throttling). The 8-bit conversion (>>8) is wrong for this controller.

**Result**: FIXED. Audio is clean on Mac. Logs confirm: 16,001 B/s steady, pacing=on, partial=0, odd=0.
