# Devlog — 2026-02-25

## Fix: Bluetooth auto-reconnect + HID on first boot

### Problem
Two related issues after device reboot:
1. `auto_reconnect.sh` always fails with **"Protocol not available"** — device never reconnects to paired hosts
2. Even when manually connected, **buttons don't work on first connection** — must disconnect and reconnect for HID to function

### Investigation

**Initial hypothesis (disproven):** CPU contention from Python 3.9 cold-start (`menu_ui.py`) starving D-Bus calls. Tested by reordering boot sequence. Result: same "Protocol not available" error. The menu UI was a red herring.

**Real root cause #1 — `Role=server` blocks outbound connect:**
Both `mic_bridge.c` and `hid_keyboard.c` registered profiles with `Role = "server"`. In BlueZ 5.55, `Device1.Connect` skips profiles locked to server role — they only accept incoming connections. With both profiles server-only, BlueZ found zero connectable profiles.

**Real root cause #2 — `Device1.Connect` only connects HFP, not HID:**
After fixing Role, `Device1.Connect` succeeded but only established HFP (outbound RFCOMM to the Mac's AG). HID never connected because `Device1.Connect` matches profiles against the remote device's UUIDs — the Mac advertises HFP-AG (0x111f) but not HID Host. macOS only connects HID L2CAP channels when *it* initiates the connection, not when the device initiates via `Device1.Connect`.

### Fixes

**Fix 1: Remove `Role = "server"` from both profiles**
- `daemon/mic_bridge.c`: Removed `Role = "server"` dict entry from `register_hfp_profile()`
- `daemon/hid_keyboard.c`: Removed `append_dict_entry_string(&dict, "Role", "server")` from `register_hid_profile()`
- Without explicit Role, BlueZ allows both inbound and outbound connections

**Fix 2: Outbound HID L2CAP connect (device-initiated HID reconnect)**
- `daemon/hid_keyboard.c`: Added `connect_hid_to_host()` — connects outbound L2CAP to host's PSM 17 (Control) + PSM 19 (Interrupt). Triggered by 7-byte IPC message `[0xFF, bd_addr[6]]` on `/tmp/spotifone_hid.sock`.
- `scripts/auto_reconnect.sh`: After `Device1.Connect` succeeds, sends the HID connect IPC via Python one-liner. Includes 1s delay for ACL link to stabilize.
- This implements the HID spec's "HID_Device Reconnect" flow (section 5.4.1)

**Fix 3: Boot order (minor)**
- `scripts/bt_init.sh`: Moved `menu_ui.py` startup to after auto-reconnect (Step 12). Not the root cause, but cleaner ordering.

### Verification
Deployed, rebuilt on device, rebooted. Results:
```
auto_reconnect.log:
  SeanAir15 connected successfully (attempt 1/5)
  SeanAir15 HID connect triggered

hid_keyboard.log:
  Outbound HID connect to 1C:F6:4C:90:5B:CB
  Control channel connected (PSM 17, outbound)
  Interrupt channel connected (PSM 19, outbound)

mic_bridge.log:
  PTT START / PTT STOP  ← buttons working on first connection
```

User confirmed: auto-reconnect works on boot, and buttons work on first connection without disconnect/reconnect cycle.

- #49: `Device1.Connect` only connects profiles with matching remote UUIDs — HID requires device-initiated L2CAP to host's PSM 17/19
