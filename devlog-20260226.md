# Devlog — 2026-02-26

## Bug regression: first-boot buttons still don't work

### Symptom
Same as last night — first connection after reboot: buttons don't work. Manual
disconnect/reconnect from macOS: buttons work immediately.

### Root cause analysis

**Root cause #1 (primary): "already connected" path skips HID trigger**

`auto_reconnect.sh` checks `Connected` property before calling `Device1.Connect`.
If macOS auto-reconnected HFP before the script ran, `Connected=true` is already set.
The script logs "already connected" and breaks — without ever sending the
`[0xFF, bdaddr]` IPC to `hid_keyboard`. HID channels are never established.

Last night this didn't trigger because the device was freshly turned on and macOS
hadn't reconnected yet. Today, macOS was faster (likely already awake with BT active),
so it reconnected HFP on its own before `auto_reconnect.sh` ran the check.

macOS auto-reconnects HFP (it's the audio gateway, remembers the HF device). It does
NOT auto-reconnect HID because the HID channels were established device-initiated last
time (our outbound L2CAP to macOS PSM 17/19). macOS doesn't replicate device-initiated
connections on reconnect — only host-initiated (NewConnection) ones are remembered.

**Root cause #2 (secondary): PSM19 failure leaves g_ctrl_fd open**

In `connect_hid_to_host()`, if PSM17 connects but PSM19 fails:
- `g_ctrl_fd` is set to the PSM17 socket (assigned before PSM19 attempt)
- `g_intr_fd` stays `-1` (PSM19 local fd is closed, but g_ctrl_fd is not reset)

This orphaned `g_ctrl_fd` confuses subsequent `NewConnection` callbacks: macOS's
PSM17 connection hits `g_ctrl_fd < 0` → FALSE → falls into `g_intr_fd < 0` → TRUE
→ macOS PSM17 (control channel) is wrongly assigned as `g_intr_fd`. HID reports go to
the wrong channel. Buttons still don't work, now in a more confusing way.

### Fixes

**Fix 1: `scripts/auto_reconnect.sh`**
Added HID IPC trigger to the "already connected" path (mirrors the "connected
successfully" path exactly). Now both paths trigger `[0xFF, bdaddr]` to `hid_keyboard`
with `sleep 1` before sending.

**Fix 2: `daemon/hid_keyboard.c`**
In `connect_hid_to_host()`, PSM19 failure branch: added `close(g_ctrl_fd);
g_ctrl_fd = -1;` before returning -1. Leaves clean state (-1/-1) so subsequent
NewConnection callbacks aren't confused by a dangling ctrl socket.

### Lessons
- #50: HID trigger must fire in BOTH connection paths in auto_reconnect.sh — macOS
  may auto-reconnect HFP before the script runs, triggering the "already connected"
  early exit. Only checking "connected successfully" misses this case.
- #51: In connect_hid_to_host(), a partial connect (PSM17 OK, PSM19 fail) leaves
  g_ctrl_fd set. This corrupts NewConnection assignment order. Always clean up ALL
  global fds on any failure path.
