# Devlog — 2026-02-24

## Fix: HID keyboard not working when audio (mic_bridge) is also running

### Problem
Audio (mic_bridge) and HID keyboard (hid_keyboard) each work individually but fail when combined. When both run together, only audio works — keyboard keys never arrive at the Mac.

### Root Causes

**Bug 1: Missing `-P input` in combined mode (`bt_init.sh:87`)**
When `KEYBOARD_FIRST=0` (normal combined mode), bluetoothd started without `-P input`. BlueZ's built-in input plugin bound L2CAP PSM 17/19 internally, preventing hid_keyboard from registering its HID profile. The flag was only present in `KEYBOARD_FIRST=1` mode — a copy-paste omission.

**Bug 2: Dual-agent conflict**
Both daemons registered their own D-Bus pairing agent and called `RequestDefaultAgent`. mic_bridge starts first and wins the slot. When hid_keyboard starts and calls `RequestDefaultAgent`, one of them loses pairing ability.

### Fixes

1. **`scripts/bt_init.sh`**: Added `-P input` to combined-mode bluetoothd startup (line 87). Updated log message.

2. **`daemon/hid_keyboard.c`**: Removed all agent registration code (~230 lines removed):
   - Removed `AGENT_PATH`, `AGENT_IFACE` constants
   - Removed `AGENT_INTROSPECT_XML`
   - Removed `agent_message_handler()` function
   - Removed `register_agent()` function
   - Removed agent vtable, object path registration, and cleanup in `main()`
   - mic_bridge's agent handles all pairing for both profiles

### Verification
- Code reviewed for correctness (no orphaned references)
- File reduced from 881 → 652 lines
- Needs on-device test: deploy, reboot, re-pair, verify both audio + keyboard work

### Lessons
- #39: `-P input` needed in ALL modes, not just keyboard-first
- #40: Only one daemon should register a BlueZ pairing agent
